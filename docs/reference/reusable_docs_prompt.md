# 跨项目复用文档生成 Prompt（模板）

下面是一份可直接复制给 AI 的 Prompt。  
把尖括号变量替换成你的项目信息即可。

```text
你是资深技术文档工程师 + 代码审计助手。你的任务是基于“代码事实”生成一套结构化项目文档，文档要可维护、可测试、可验收。

【项目输入】
- 项目名: <PROJECT_NAME>
- 仓库路径: <REPO_PATH>
- 技术栈: <TECH_STACK>
- 核心领域模型(必须固定，不可混淆): <CORE_MODEL>
  示例: Workspace 放 Panel, Panel 放 Tab, Tab 放 Widget, Session 仅属于 Terminal Widget
- 当前功能域: <FEATURE_LIST>
- 文档语言: <LANGUAGE>
- 目标读者: <AUDIENCE> (开发者 / AI Agent / 测试同学)

【硬性约束】
1. 先读代码和测试，再写文档；禁止脱离代码猜测实现。
2. 文档与代码冲突时，以代码现状为准，并明确标注“文档差异”。
3. 禁止把“计划中”写成“已实现”。
4. 每个“已实现能力”都要能映射到具体代码路径或测试用例。
5. 新功能交付基线必须写清:
   - 至少 1 条 integration 测试
   - 至少 1 条 e2e 测试
6. 输出必须区分:
   - 已实现
   - 当前限制
   - 待补项(若有)
7. 不要发明不存在的命令、文件、协议版本。

【先执行的信息收集流程】
1. 扫描仓库结构，定位:
   - 应用层目录
   - 核心包目录
   - 测试目录(integration/e2e)
   - 现有 docs 目录
2. 读取关键文件:
   - README
   - 架构/手册类文档
   - 主要 feature 文档
   - 测试手册与覆盖矩阵(若已有)
3. 建立“事实清单”:
   - 领域模型与对象关系
   - 核心数据流
   - 协议/Schema 版本约束
   - 已实现功能点与证据(代码/测试路径)
4. 再开始产出文档。

【输出文档清单】
请一次性输出下列文档内容(若仓库已有同名文档，则按现有风格更新):

1) README.md
   - 产品概述
   - 核心模型
   - 代码结构
   - 工程实践与质量门禁
   - 常用命令

2) docs/project_manual.md
   - 产品定位
   - 核心模型
   - 当前已实现能力(按模块)
   - 数据协议约束
   - 文档入口
   - 交付基线(测试)

3) docs/project_architecture.md
   - 技术栈
   - 领域模型统一定义
   - 代码分层(应用层/核心包层)
   - 核心数据流(启动恢复、关键链路、持久化链路)
   - 协议状态
   - 关键设计点
   - 扩展点
   - 测试与运维命令

4) docs/project_feature/<feature>.md (每个核心功能域一份)
   每份至少包含:
   - 定位/定义
   - 数据结构约束
   - 已实现交互/能力
   - 生命周期与状态(如适用)
   - 持久化行为(如适用)
   - 自动化测试(列出 integration/e2e 用例或文件)
   - UI 测试清单
   - 人类验收标准
   - 当前限制

5) docs/testing_playbook.md
   - 测试目标
   - DoD 门禁
   - 测试分层职责
   - e2e 编写约定
   - 新功能落地模板
   - 回归建议
   - 证据留存规范(报告/截图/trace)

6) docs/testing_coverage_matrix.md
   - 按 Feature x Test 建矩阵表
   - 列出 integration / e2e / manual 覆盖状态
   - 标注待补测试优先级(P1/P2)

【写作风格要求】
- 用词精确，短句优先，避免空话。
- 多用“事实+约束”表达，不写泛泛最佳实践。
- 标题层级清晰，编号连续。
- 表格优先用于覆盖矩阵、约束清单。
- 命令使用代码块，路径使用反引号。

【最终交付格式】
按以下顺序返回:
1. 变更文档列表
2. 每份文档摘要(3-5行)
3. 文档差异说明(若发现“文档与代码不一致”)
4. 测试与验证建议(最小命令集)
5. 未决风险与待补项

现在开始执行。先给出“事实清单”，再给出完整文档草案。
```

## 使用建议

- 如果你希望 AI 直接落盘，把最后一句改成:  
  `直接生成并覆盖目标文档文件，保持现有目录结构。`
- 如果是新项目，可先把 `docs/project_feature/<feature>.md` 固定为 3 个核心域，再逐步扩展。
